# LLM 中的 Token 详解

## 1. 大模型的基本工作原理

> 大模型的本质

大模型本质上是一个**巨大的数学函数**，具有以下特点：

- 内部全是矩阵运算
- 输入的是数字，输出的也是数字
- 根本不理解人类的文字

----



> Tokenizer 的角色

在人类和大模型之间必须有一个**翻译官**，这就是 **Tokenizer**。它负责两个核心功能：

- **编码（Encoding）**：把文字转换为数字

- **解码（Decoding）**：把数字转回文字






## 2. Tokenizer 的训练过程

**业界常用的 Tokenizer 实现算法**：

- **Unigram**：Google 常用
- **BPE（Byte Pair Encoding）**：OpenAI 和 Anthropic 常用
  - BPE 算法的核心思想：**从训练材料中找出经常在一起使用的字，把它们合并作为一个 Token**




**BPE模型训练大致原理**

<img src="assets/image-20260131172703315.png" alt="image-20260131172703315" style="zoom:80%;" />

**训练产物**：BPE Tokenizer 训练完成后得到两个核心组件

1. **词表**：记录 Token 和 Token ID 的映射关系
2. **合并规则**：记录哪些 Token 可以合并成新的 Token





## 3. Tokenizer 的使用过程

<img src="assets/image-20260131173116835.png" alt="image-20260131173116835" style="zoom:80%;" />

- **编码（Encoding）**包含两个步骤：**切分** 和 **映射**

- **解码（Decoding）**只需要**映射**这一步：
  - 模型输出 Token ID：`36`
  - 查词表找到对应的 Token：`36 → 喜欢`
  - 输出文字：`喜欢`



> 完整示例

```
输入："马克喜欢人工智能吗？"（9个字）
    ↓ 切分
Token: [马克, 喜欢, 人工智能, 吗]（4个Token）
    ↓ 映射
Token ID: [35, 36, 37, 38]
    ↓ 送入模型
模型输出: 36
    ↓ 映射
Token: 喜欢
    ↓ 输出
"喜欢"
```





## 4. Token 与字数的换算关系

> 换算比例

| 语言     | 换算关系                      |
|----------|-------------------------------|
| 中文     | 1 Token ≈ 1.5 - 2 个汉字      |
| 英文字母 | 1 Token ≈ 4 个字母            |
| 英文单词 | 1 Token ≈ 0.75 个单词         |



> 实际应用示例

以 **GPT-4 的 40 万 Context Window** 为例：

- **中文**：40万 Token ≈ **60-80万个汉字**
- **英文**：40万 Token ≈ **30万个单词**



> 为什么 Token 不等于字数？

- 因为 Tokenizer 不仅是**翻译机**，还是**压缩机**：

  - 经常在一起出现的字会被合并成一个 Token

  - 例如："马克喜欢人工智能吗"（9个字）→ 4个Token

  - 压缩率：约 44%


- 这种压缩机制使得：

  - 模型的输入更少

  - 训练和推理效率更高

  - Context Window 能容纳更多实际内容






## 总结

> 核心概念

1. **Token**：Tokenizer 切分后的最小单位（文字形式）
2. **Token ID**：Token 对应的数字编号（数字形式）
3. **Token 和 Token ID 是一对一关系**，就像硬币的正反面



> Tokenizer 的价值

- **翻译功能**：实现文字与数字的双向转换
- **压缩功能**：将常见词组合并，减少输入长度
- **提升效率**：减少模型计算量，加快推理速度



> 关键要点

✅ Token ID 只是编号，不含语义信息  
✅ 合并后的词可以继续合并  
✅ 1个 Token ≈ 1.5-2个汉字 ≈ 0.75个英文单词  
✅ Tokenizer 是被训练出来的，不是手工编写的规则
