# Context Engineering 详解

> 基于视频：[Context Engineering：概念与技术实现深度解析](https://www.youtube.com/watch?v=25DEMZ7wsSM)



## 1. Context 和 Context Window 介绍

### 1.1 Context 的定义

Context（上下文）：从外部来看，大模型就像一个函数，你给它输入，它就会给你输出。这里的**Context**指的就是模型的输入，包括：

```bash
- 用户问题
- 背景信息
- 相关资料
- 可用工具列表
- 工具执行结果
- 历史对话
```

模型把这些输入作为上下文，并基于这些内容来生成答案。





### 1.2 Context Window

**Context Window** 是指模型的输入中最多能包含多少个 **token**。

> **Token** 的概念

- Token 是文本被拆分成的最小单位
- 可能是一个字、一个词或一个标点符号
- **1个 token ≈ 0.75个英文单词 ≈ 1.5个汉字**



> 主流模型的 Context Window 对比

| 模型 | Context Window | 相当于容量 |
|------|---------------|-----------|
| Gemini 2.5 Pro | 100万 token | 约7本书 |
| GPT-5 | 40万 token | - |
| Claude 4 | 20万 token | - |
| DeepSeek V3 | 12.8万 token | - |

> **注意**：一旦输入超过Context Window限制，前面的内容会被丢弃，只保留最后的部分。





### 1.3 为什么不能直接把所有资料都放进Context？

虽然现代模型的Context Window很大，但在实际使用中会遇到三个核心问题：

- **问题1**：大多数模型的Context Window有限

  - Gemini 2.5 Pro的100万token算是很大的了

  - 大部分模型的Context Window远小于此

  - 为了节省成本，小模型的Context Window可能只有几万token

  - 可能连一个产品使用手册都装不下


- **问题2**：输入太杂乱会影响模型理解，即使没有超过Context Window上限
  - 如果输入内容：杂乱无章、信息冗余、内容矛盾
  - 模型可能会：混淆重点，输出含糊其辞的回答
- **问题3**：输入越多，成本越高
  - 大语言模型按token计费
  - token数量越多，消耗越大
  - 不加控制地塞入大量上下文会带来不必要的开销
  - **优化输入 = 优化成本**



**解决方案**：与其把所有信息都塞进去，不如经过精心的设计和组织，确保模型看到的是准确、有结构、重点突出的内容。





## 2. Context Engineering 介绍

### 2.1 什么是 Context Engineering？

**Context Engineering**（上下文工程）是对模型输入进行优化的技术体系。

- **核心思想**：不改变模型结构，只改变模型看到什么
- **目标**：让模型在有限的Context Window里尽可能：
  - **理解得更准**
  - **答得更好**
  - **花得更少**



### 2.2 Context Engineering 为什么最近才火起来？

虽然Context这个概念一直都存在，但Context Engineering这个词是最近才火起来的，主要有两个原因：

- **原因1**：现在的模型已经足够强大

  - 与一两年前相比，现在的模型强大了不止一点半点

  - 只要给到足够精细的要求和足够完整的相关资料，模型就能给出非常准确有用的回答

  - **如果模型回答不满意，往往不是因为模型不够强，而是因为没有给到模型足够清晰和完整的信息**


- **原因2**：Agent 的兴起，**Agent**是一种把大模型和工具结合起来，让模型能够独立感知环境、影响环境，从而解决用户问题的技术。



Agent运行涉及的Context内容：

- 可用工具信息
- 工具执行结果
- 使用多少次工具就会有多少个工具执行结果

<img src="assets/image-20260131175905764.png" alt="image-20260131175905764" style="zoom:67%;" />

Agent运行时间长了，这些结果会占满整个Context Window，**Context的管理效果会直接影响到Agent的执行结果**，这就是为什么Agent兴起后，Context Engineering的话题也随之火了起来。



### 2.3 Context Engineering 的定位

Context Engineering并不是某项单一的技术，而是**由多种技术组成的方法体系**。

它关注的不是怎么训练模型，而是**怎么精心设计给模型的输入内容**。





## 3. Context Engineering 的实现方法

基于LangChain的设计框架，可以将Context Engineering划分为四大类：

1. 保存 Context
2. 选择 Context
3. 压缩 Context
4. 隔离 Context





### 3.1 保存 Context

> 定义

把Context做筛选、总结，并找个地方保存起来（如内存、硬盘），在模型需要的时候再发给模型。



> 典型案例：ChatGPT的记忆功能

**示例场景**：

```
用户："我的名字是马克，我有一个频道叫做马克的技术工作坊，
      在这个频道里面我分享过关于Agent、RAG、MCP、A2A等相关的内容。"
      
系统：Updated saved memory ✓
```

- ChatGPT会把这段内容存入到记忆库
- 可以在记忆库中查看已保存的记忆
- 这解决了**信息持久化**的问题

**但是**：光存下来还不够，如何从庞大的记忆库里精准找到所需信息？这就引出了下一个技术。





### 3.2 选择 Context（最核心）

> 定义

从海量信息中选择出一部分与用户问题最相关的内容，并把它们放在模型的Context里。

**一个好的选择策略是整个系统高效准确运行的保障。**



> 静态选择

- **定义**：把一些永远重要、必须遵守的信息，在每一次请求时都全部放到Context里。

- **特点**：

  - 就像给AI焊在脑子里的系统指令或核心原则

  - 无论用户问什么，这些信息都必须在场


- **典型案例**：

  - **Cursor的rules文件**

  - **Claude Code的CLAUDE.md文件**


- 这些文件指定了：

  - 当前项目信息

  - 编码时需要遵守的规范

  - 其他至关重要的信息


**策略**：全都要！只要确保它们加起来不会撑爆Context Window就行。

----



> 动态选择

- **定义**：选择与用户问题最相关的内容，并把它放入Context中。

- **特点**：

  - 不是把所有东西都塞进去

  - 像一个聪明的图书管理员，为你精准地找到需要的那几页


- **典型案例**：

  1. **ChatGPT从记忆库中挑选相关记忆**
     - 从大量记忆中挑选与用户问题最相关的部分


  2. **工具选择**
     - 从包含数十个甚至数百个工具中
     - 挑选与用户问题最相关的3个工具
     - 放到模型的Context中


- **最著名的实现方式：RAG（检索增强生成）**，RAG通过向量检索等技术，从大量文档中找到与用户问题最相关的片段，动态地放入Context中。





### 3.3 压缩 Context

> 背景问题

Agent运行过程中会在Context里积累大量的历史消息：
- 模型的输出文本
- 工具的执行结果

如果不做处理，这些历史消息会迅速挤满Context的全部空间，从而影响模型的后续回答效果。

----



> 解决方案：压缩历史消息

**典型案例：Claude Code的auto-compact**

Claude Code是一个用于写代码的Agent，运行过程中会：

<img src="assets/image-20260202105336508.png" alt="image-20260202105336508" style="zoom:80%;" />

**处理机制**：当Context Window使用量超过**95%**时，运行**auto-compact**程序，对以往的Context做压缩（总结之前的内容），把原来的详细内容扔掉，Context Window使用量降下来

**高级功能**：用户可以在`CLAUDE.md`中指定压缩方案，例如：

```markdown
压缩时重点保留：
- 测试输出
- 代码改动方面的内容
```





### 3.4 隔离 Context

> 定义

不同模块之间的Context是**互相隔离的、互不干扰**。这通常发生在**Multi-Agent系统**里面。

----



> 典型案例：Anthropic的Multi-Agent研究系统

**系统架构**：

<img src="assets/image-20260202110037542.png" alt="image-20260202110037542" style="zoom:80%;" />

**关键特征**：
- **Lead Agent**：负责任务下发和归纳总结
- **Subagent**：各自负责特定任务
  - 有自己独立的工具
  - 有独立的运行历史
  - 有独立的记忆体系
- **Context隔离**：各个subagent的Context互相隔离、互不影响

**优势**：1) 职责清晰; 2) 彼此之间互不干扰; 3) 避免了Context的混乱和冗余





## 4. 总结

### 4.1 核心概念回顾

| 概念 | 定义 |
|------|------|
| **Context** | 模型的输入（问题、资料、工具信息、历史对话等） |
| **Context Window** | 模型输入的容量上限，以token数量为单位 |
| **Context Engineering** | 精心设计给模型的输入内容，让模型在有限的Context Window里理解得更准、答得更好、花得更少 |





### 4.2 Context Engineering 四大实现方法

```
┌─────────────────────────────────────────┐
│      Context Engineering 方法体系        │
├─────────────────────────────────────────┤
│ 1. 保存 Context                         │
│    └─ 持久化信息，需要时调用             │
│                                         │
│ 2. 选择 Context（核心）                 │
│    ├─ 静态选择：永远重要的信息           │
│    └─ 动态选择：相关性检索（如RAG）      │
│                                         │
│ 3. 压缩 Context                         │
│    └─ 总结历史，释放空间                │
│                                         │
│ 4. 隔离 Context                         │
│    └─ 模块独立，互不干扰                │
└─────────────────────────────────────────┘
```





### 4.3 Tips

Context Engineering并不是某个特定而具体的技术，而是**某一类技术的统称**，包含的技术五花八门。

## 5. 实践建议

在构建AI系统时，应该：

1. ✅ **精心设计输入**：不要简单地把所有资料都扔给模型
2. ✅ **合理选择信息**：根据重要性和相关性筛选Context
3. ✅ **控制输入大小**：在效果和成本之间找到平衡
4. ✅ **优化信息结构**：确保内容准确、有结构、重点突出
5. ✅ **动态管理Context**：根据任务需要保存、选择、压缩、隔离

记住：**优化输入 = 优化效果 = 优化成本**
